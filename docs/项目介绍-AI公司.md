# AI 公司:一个可以被像真人一样管理的 agent 群



**做一个能像管理真人公司一样管理的 AI 群——每个 AI 是聊天软件里独立的好友账号,可私聊可拉群;你是 CEO,用自然语言派活;主管 AI 拆任务、带 sub-agent、对你汇报;能开会、能执行、能复盘、能成长。**

不是又一个 "AI 助手",不是又一个 "agent 框架",是一个**有组织、有身份、能干活的 AI 团队**,你以"老板"的姿态用它,而不是"提示词工程师"的姿态。

---

## 为什么这件事现在值得做

1. **API 不再是瓶颈**——2026 年 function calling / MCP / 长上下文 / 流式都成熟了。**真正的瓶颈是组织方式**:把 10 个 agent 怎么协调出 1 个人 10 倍的产出,而不是 10 个吵成一锅粥。
2. **市面上 agent 框架都在做"内核"**(AutoGen、LangGraph、CrewAI、MetaGPT)——它们都是**程序员的玩具**:agent 是进程里的对象,UI 是开发者自己拼的,普通人摸不到。
3. **产品形态的空位**:**让普通人能像加微信好友一样添加 AI、像拉群一样调用 AI 团队**——这件事没有任何成熟产品在做。Character.AI 类有角色没编排,AutoGen 类有编排没产品形态。

---

## 产品长什么样(用户视角)

```
[用户]  在 Telegram(开发期) / QQ(生产) 上:

  → 加好友:"小K"(产品经理 AI)、"老张"(架构师 AI)、"莉莉"(用户研究 AI)...
     每个 AI 是独立账号,有头像、有人设、可单独私聊。

  → 拉群讨论:"我想做个 XX,帮我看看"
     ↓
  → 主持人 AI 自动跳出:"这个事可以分三个维度——市场、技术、用户体验,
     我让老张、莉莉、小K 各负责一个。"
     ↓
  → 三个 AI 各自展开,轮流发言,相互质询。
  → 你随时插话:"等等,技术那块预算多少?" → 流程被你打断、转向。
     ↓
  → 主持人收尾,给一份结论。
  → 复杂任务时,某个 AI 会进一步拉自己的"私群"(sub-agent 团队)做细化,
     做完汇报给你——你看不到内部细节,但能跳级抽查。

  → 涉及执行(写代码/查数据/发邮件):
     • 可逆动作 → AI 自己干
     • 中等动作 → 部门主管批
     • 重大不可逆 → 弹给你批
     • 危险动作 → 硬禁止
```

不是"调用 API 的网页 demo",是**聊天软件里活的同事**。

## 三个别人没做对的事(护城河)

详细对比见[原方案 §12](方案-AI微信群-需求与技术方案.md#12-同类项目对比与选型)。摘要:

### 1. 真人当 CEO,信用分配是局部的

MetaGPT / ChatDev 这类"AI 软件公司",CEO 也是 agent,整条流水线全自动跑完——结果就是**没人能评价它做得好不好**,只能看最终交付物,中间过程黑箱。

我们让**真人当 CEO**,但**只评直属主管,看不到 sub-agent**;主管负责评自己的下属。这把"功劳算谁"这个扁平多 agent 系统的死结**剪开成局部问题**。这是组织设计上的关键洞察。

### 2. 否决"裁判打分+末位淘汰",选择保护多样性

2024-2025 这两年,"agent 自我进化"项目的标准做法是:让 AI 互评打分、低分淘汰。**这个方向系统性地错了**:

- LLM-as-judge 有偏见、会自我循环;
- stack ranking 会**最先淘汰最该留的唱反调者**——因为它们最不"合群";
- 最终你养出一群应声虫,把"多视角"这个多 agent 系统**唯一的红利**毁掉。

我们的方案:**Reflexion 记忆**(每场复盘存教训,下次召回)+ **多样性保护**(异见席位免死、信誉做权重不做死刑)。这不是更保守,是**想清楚了不能那样做**。

### 3. 执行可验证 = 成长机制一直缺的"客观奖励"

讨论类任务没有标准答案,只能 AI 互评(见上一条的陷阱)。但**执行类任务有**:代码跑没跑通?测试过没过?规格达没达?**一旦把执行接进来,客观反馈白送上门。**

这就形成闭环:**讨论给判断 → 执行给事实 → 事实校准判断**。多数项目要么纯讨论(辩论类)要么纯执行(SWE-agent 类),不把两者咬合。

### + 一个产品形态上的护城河

**聊天平台原生多身份**——每个 AI 一个独立真账号,可私聊可拉群,混在真人通讯录里行为像真人。这不是技术问题,是**身份与平台适配问题**,开源编排框架基本不解决。我们扛这部分的工程,别人就追不上。

## 当前进度(不是 PPT 项目)

不是空想,有可工作的代码和打透的设计。

**已经跑起来的**:
- 一个**圆桌讨论引擎**——planner / moderator / synthesizer 三角色,动态生成讨论维度,流式输出,支持 markdown + LaTeX 渲染。代码量小、可读、能改。
- FastAPI + 简洁 web UI,本地一行命令起服务。
- 兼容任何 OpenAI 协议后端(目前用 opencode go / deepseek-v4-pro 实测)。

**已经验证的关键技术风险**:
- ✅ 后端 function calling 完整支持(流式非流式都行,工具决策 3-4s 不慢)。
- ✅ 折叠 system 消息进 user 消息能跑通(后端拒收 system 的特殊性已绕过)。
- ✅ 流式输出避开了长推理的 idle 断连问题。

**已经想透的设计(全部落档)**:
- §3 需求与风险、§4 关键决策(身份/调度/平台)
- §5 完整技术架构(适配器层、引擎、调度器、持久层)
- §7 组织层级化(CEO / 主管 / sub-agent、信用分配、加层触发、评价向下渗透机制)
- §8 成长机制(Reflexion + 多样性保护、为什么否决末位淘汰)
- §9 执行层(ReAct + MCP + 沙盒、四级审批阶梯、混合定级防奖励黑客)
- §12 行业对位与选型(我们 vs AutoGen / LangGraph / MetaGPT / CrewAI)

**这意味着什么**:技术地基稳了,产品差异点也想透了——剩下是工程活,不是研究活。

## MVP:第一刀切在哪

```
Step 1  把现有圆桌引擎从"流水线"改成"调度循环"   [纯本地,web 验]
Step 2  加持久层(好友/群/历史/Reflexion 记忆)     [SQLite 起步]
Step 3  加"人在环打断"——任何时候插话改方向         [纯本地]
Step 4  接 Telegram 适配器——产品形态首次落地       [真"AI 好友"]
Step 5  接 QQ 官方机器人——面向中文生产环境         [生产]
```

**前 3 步全在本地、平台无关、不依赖任何账号审核**。换言之:**两周左右我们就能拿到一个"可以在浏览器里和 AI 团队开会、它们能记住你"的 demo**。这是好早期项目应该有的节奏——不下注未审就跑通的东西。

Step 4 一旦跑通,产品形态就立住了——剩下的都是规模、平台、商业的事。

## 选型

**编排底座**:倾向**混合**——MVP 用现有自研引擎快速验证产品形态;做到层级递归和执行循环时,把调度器层换成 **AutoGen** 或 **LangGraph**(具体哪个待最小原型对比)。**我们不在调度内核上和大厂卷**,护城河在组织治理和产品形态。

**平台选择**:
- 开发期 **Telegram**——原生多 bot、无配额、免费、能验全部产品形态。
- 生产 **QQ 官方机器人**——面向中文用户、合规、每个 bot 独立账号(企业微信做不到这个,一个应用=一个身份,所以被否决了)。

**关键已知风险**:QQ 官方主动消息有配额——圆桌要大量主动刷消息,生产能不能撑住是头号未知,得在 Telegram 验逻辑、迁 QQ 时实测。
