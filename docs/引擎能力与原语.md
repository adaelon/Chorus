## 1. 一句话模型

**配方 = 原语在共享状态 `GroupState` 上的一张有向图（DAG）**。引擎只提供原语（节点）+ 编译/校验/运行；
"圆桌/扇出/auto/自定义"都是同一套原语的不同接线（§6.6）。图是**声明式 JSON**，可存库、校验、渲成卡片流、跑。

```
START → [原语节点]…(条件边/循环)… → SYNTHESIZE/PRODUCE → END
```

## 2. 共享状态 GroupState（orchestrator/app/state.py）

所有原语在这**一份扁平状态**上读写（⚠ 无"子群/嵌套"概念，见 §10 边界）。

| 字段 | 类型 | 含义 |
|---|---|---|
| `group_key` | str | 会话键（= LangGraph thread_id） |
| `roster` | list[AgentSlot] | 到场成员（**单一平铺名单**），AgentSlot=`{contact_id, dimension}` |
| `history` | list[Msg] | 群历史（单条线），Msg=`{sender_id, sender_kind(human/ai/moderator), text, dimension, ts}` |
| `candidates` | list[Candidate] | 扇出候选，Candidate=`{contact_id, dimension, text}` |
| `picked` | list[Candidate] | 人工策展选中 |
| `claims` | list[Claim] | 点账本（圆桌远场记忆），Claim=`{speaker_id, text, turn}` |
| `output` | str\|None | SYNTHESIZE 的最终产出 |
| `turns_since_human` | int | 距上次真人发言的轮数（圆桌预算闸计数） |
| `max_turns_per_human` | int=6 | 圆桌预算闸上限 |
| `pending_human` | Msg\|None | 异步注入的真人插话（待 human_gate 消化） |
| `next_speaker` | str\|None | 下一发言人 contact_id（router 写、turn 读） |
| `next_decision` | str\|None | **路由信号**（router/human 写、紧跟的条件边读） |
| `directed_queue` | list[str] | @定向插话队列（§6.20）：human_gate 填、schedule 按序 pop，跳过主持人/预算闸 |
| `directed_active` | bool | 当前 turn 是否定向修订（schedule 写、turn 读作"真人点名要你修改"框架，§6.20） |
| `stop_reason` | str\|None | 停止原因（budget/plan_budget/moderator…） |
| `plan_steps` | int | auto 配方已组原语步数（步数闸计数） |
| `max_plan_steps` | int=8 | auto 步数闸上限 |
| **执行层（S11–S16，见 §9）** | | |
| `pending_tools` | list[ToolCallIntent] | 已闭合待执行的工具意图（llm_plan 写、tool_dispatch 消费） |
| `tool_results` | list[ToolResult] | 工具执行结果（成功/失败都入） |
| `agent_steps` | list[AgentStep] | ReAct scratchpad：历次"工具+参数+结果"（planner 据此续推，§6.24） |
| `turn_traces` | list[TurnTrace] | 工具化发言 trace，按 (speaker,turn) 归属（抽屉/drill-in） |
| `trace_events` | list[TraceEvent] | 执行子图节点级审计事件 |
| `run_status` | running\|aborted\|failed\|degraded\|done | 执行 loop 运行态（降级/取消可观测） |
| `retry_budget` | RetryBudget | 工具重试预算 |
| `abort_requested` | bool | 入口取消信号（穿透执行 loop） |
| `sandbox_ready` | bool\|None | 沙箱探活态（down→降级边） |
| `last_tool_error` | ToolRuntimeError\|None | 最近一次工具 infra 错误 |

## 3. 原语清单（registry，orchestrator/app/recipes/spec.py）

**11 个用户可见原语**（S10a/b 加 `produce`/`deliver`）。`extract`/`generate` 是 `turn`/`fanout` 内部子函数，不入 registry；纯 `curate()` 是 `curate_gate` 的 apply 逻辑。

| 原语 | 形态 | 人话 | needs(硬前置) | reads | writes | emits(路由标签) | budget(闸) |
|---|---|---|---|---|---|---|---|
| `clarify` | human | 澄清需求 | history | history | history | — | — |
| `frame` | transform | 分配维度 | history, roster | history, roster | roster | — | — |
| `fanout` | transform | 并行出候选 | roster | roster, history, claims | candidates | — | — |
| `turn` | transform | 轮流发言 | next_speaker | next_speaker, directed_active, roster, history, claims | history, turns_since_human, claims | — | — |
| `schedule` | router | 主持人调度 | roster | pending_human, turns_since_human, max_turns_per_human, roster, claims, history, directed_queue | next_speaker, next_decision, stop_reason, directed_queue, directed_active | next_speaker / yield_to_human / stop | turns_since_human ≥ max_turns_per_human（@定向时跳过） |
| `plan` | router | 主持人现编 | roster | plan_steps, max_plan_steps, candidates, roster, claims, history | next_decision, next_speaker, plan_steps, stop_reason | fanout / speak / synthesize / stop | plan_steps ≥ max_plan_steps |
| `human_gate` | human | 真人插话窗口 | — | pending_human, history, turns_since_human | history, pending_human, turns_since_human, next_decision, directed_queue | continue / end | — |
| `curate_gate` | human | 人工策展 | candidates | candidates, picked, roster, history, claims | candidates, picked, next_decision | curate / synthesize | — |
| `synthesize` | transform | 出结论/决策 | — | claims, history, picked, candidates | output | — | — |
| `produce` | transform | 出产物 | — | claims, history, picked, candidates | output | — | — |
| `deliver` | human | 出产物形态选择闸 | — | — | next_decision | decide / produce | — |

> 注：所有原语都只读写**同一个平铺 roster**——没有任何原语能"把成员分组、各组先开会、再上提"。这是 §10 边界的根。
>
> **`turn` 工具化模式（S11–S16，§6.24，见 §9）**：注入 `plan_stream`+`execute`（+可选 `tool_gate`）时，`turn` 发言前会先跑一段 ReAct **工具阶段**（沙箱/MCP/内置工具），把发现拼进发言 context；未注入=纯发言（门控，A3）。工具阶段是 `turn` 节点**内部**的执行子图，**不**是 DAG 上的原语——配方层看不到它，故原语表的 reads/writes/emits 不变（执行细节见 §9）。

## 4. 原语三态（§6.16 维度一）

| 形态 | 行为 | 出边 | 编译规则 |
|---|---|---|---|
| **transform** | 纯 `state→state delta` | 单条普通边 | `add_edge` |
| **router** | 写 `next_decision`，不暂停 | 条件边（多分支） | `add_conditional_edges`(读 next_decision) |
| **human** | `interrupt()` 暂停等人，恢复后写 state | 条件边（"会暂停的 router"） | interrupt 节点 + 条件边 |

铁律：**原语只写 state、绝不决定"去哪"；路由 100% 在边上**。预算闸（router 的 `budget`）声明式进 spec，编译器自动据此插闸。

## 5. 声明式 DAG 格式（配方 JSON）

```json
{
  "recipe": "<id/slug>", "version": 1,
  "nodes": [ {"id": "<实例名>", "use": "<原语名>", "args": null} ],
  "edges": [ {"from": "START|节点id", "to": "END|节点id", "when": <条件>?} ]
}
```
- `id` 与 `use` 分开：一张图可用同一原语多次（不同 id）。
- 边：无 `when`=普通边；带 `when`=条件边（按 from 归组）；无 `when` 的那条作 **else** 兜底。
- `args` 当前未用（spec.args 全 None，留后）。

### 边条件文法 `when`（orchestrator/app/recipes/cond.py，**数据非代码、无 eval**）
```
when = {field, op, value}                  # 原子（field 限 GroupState 字段）
     | {"all": [when, ...]} | {"any": [...]}# 复合（可嵌套）
op ∈ ==  !=  >  >=  <  <=  in  empty  truthy
```
`on:label` 是 `{field:"next_decision",op:"==",value:label}` 的语法糖。

## 6. 编译与校验

- **`compile_recipe(graph, checkpointer, *, deps)`**（recipes/compile.py）：JSON 直译成 LangGraph `StateGraph`。
  节点经 `inspect` 过滤注入 deps（assign/generate/persona_provider/extract/pick/planner/compose/compose_produce/assess/reputation_adjuster）+ 据 spec.budget 自动插闸；START/END 串接。
- **`validate_recipe(graph) -> list[str]`**（recipes/validate.py，返回人话错误）四关：
  1. **needs 可达**：每节点 needs 在所有到达路径上被上游 writes（或初始输入 group_key/roster/history）覆盖（must 数据流定点）。
  2. **必有 else**：有条件出边的节点必有唯一无 when 兜底边。
  3. **环上有闸**：去掉带 budget 的 router + human 节点后须无环（human 靠 interrupt 有界，亦算闸）。
  4. **when 合法**：字段/算子白名单。
  写库（POST/PUT /recipes）与画布实时校验都复用它。

## 7. 配方库（orchestrator/app/db + service.py）

- `Recipe(id, name, builtin, graph)` 表；内置幂等 seed、不可删/改。
- 六内置配方（recipes/builtin.py）：

| id | 形状 |
|---|---|
| `roundtable` | CLARIFY→FRAME→SCHEDULE⇄TURN→HUMAN_GATE→…→SYNTHESIZE（人在环，**出结论**） |
| `roundtable_produce` | 同 roundtable，末端 SYNTHESIZE→PRODUCE（**出产物**，§6.21/S10a） |
| `roundtable_deliver` | 同 roundtable，human_gate end→DELIVER 选择闸→{SYNTHESIZE\|PRODUCE}（**结束再定**，§6.21/S10b） |
| `roundtable_continuous` | roundtable 去掉 human_gate（自动连续，到预算闸停） |
| `fanout` | CLARIFY→FRAME→FANOUT→CURATE(自循环)→SYNTHESIZE |
| `auto` | FRAME→PLAN⇄{FANOUT\|TURN}→…→SYNTHESIZE（L3 运行时即兴） |

> **圆桌 @定向插话（§6.20，S9a）**：`roundtable` 的 `turn` 出边是条件边——`directed_queue 非空 → schedule`（取下一个定向）/ 空 → `human_gate`。human_gate 收 resume 的 `directed:[contact_id]`（前端 chips 选，避文本解析）→ 填 `directed_queue`；schedule 队列优先 pop（跳主持人/预算闸）+ `directed_active=True`。效果：被@者按 @ 顺序各发一次修订、跑完停回人、主持人不接力；不@ 维持现状（主持人挑谁接）。relay 无真人/不带 directed → 队列恒空、行为不变。
>
> **修订去重（§6.20/§6.11，S9b）**：`directed_active` 的 `turn` 追加新发言前，先把该 speaker 的旧 `claims` 去重（supersede）——点账本（synthesize / 远场 context / 主持人都读它）只留该人最新一版。`history` 仍 append-only 留两版原文。非定向重复发言仍累积（§6.11 不变）。
>
> **产出形态（§6.21，S10a/b）**：圆桌末端三选一——`synthesize`(出结论=收敛共识/分歧) 与 `produce`(出产物=把原始 task 当生产任务书、交付那个东西本身) 同形异脑（dep 键 `compose`/`compose_produce` 区分）；`deliver`(human 纯选择闸：结束时 interrupt 问"结论/产出"，resume `{choice:"produce"|"decide"}` → 写 next_decision，路由复用 synthesize/produce、自己不产出)。配方 `roundtable`(出结论)/`roundtable_produce`(出产物)/`roundtable_deliver`(结束再定)。开场知道选前两张；结束才知道选第三张。relay 无人用前两张。

## 8. 协作分层 L1→L4 + 主要端点

| 层 | 谁决定 | 端点/入口 |
|---|---|---|
| **L1** 用户选现成 | 用户 | 首页配方卡 |
| **L2** 主持人荐现成 | LLM 选 roundtable\|fanout | `POST /recipe/select {task}` |
| **L3** AI 搭新图 | LLM 高层选择 + 确定性 assemble 出图 | `POST /recipe/auto {task,roster}`（存库） |
| **L4** 用户卡片流自拼 | 用户 | 配方页 RecipeEditor（增删卡/改边/实时校验/存） |

其它：`GET /primitives`（卡片库）、`POST /recipe/validate`、`POST /recipe/run {recipe_id,…}`（跑库内任意 DAG，SSE；圆桌三形态走它）、`/roundtable/stream`（默认圆桌起场）、`/session/{key}/resume/stream`（通用续场，按会话 recipe_id 取图，ChatPage 所有续场走它）、`/session/{key}/retry/stream`（出错断点续）、`/inbound`·`/curate`·`/synthesize`（扇出）、`/contacts` `/recipes` `/llm-backends` CRUD、`/relay/inbound`（telegram 桥）。

> **续场 payload（resume）按暂停点选字段**：clarify→`{answer}|{skip}`；human_gate→`{interject, directed?}`（§6.20 @定向）/ `{end}`（结束；roundtable_deliver 下 end→deliver 选择闸）；deliver→`{choice:"produce"|"decide"}`（§6.21）。必须非空（LangGraph 把 falsy resume 当未恢复）。

L3 产图机制（recipes/plan_recipe.py，§B2）：AI 只出 `RecipePlan{mode∈{roundtable,fanout}, clarify, human_in_loop}`，`assemble_recipe` 据此**裁内置模板**成保证合法的图。**AI 不裸写 nodes/edges**。

## 9. 执行层：工具化发言（S11–S16，§6.22–6.27）

让圆桌里的 AI 不止"说"，还能"动手"——发言前先用工具（跑代码 / 调 MCP / 联网）干活，再据此自然发言。**藏在 `turn` 节点内部**，对配方层透明（§3 注：不是 DAG 原语）。

### 9.1 一轮工具化发言的流程（β，§6.24）

```
turn ─→ [tool_gate?] ─不需要→ 直接流式发言
              │需要
              ▼
        工具阶段（隔离 sub-state，最多 max_tool_steps=6 步）
        llm_plan ⇄ tool_dispatch  ← ReAct loop
              │（planner 给 final / 无 pending / 沙箱降级 / 重复调用 → 停）
              ▼
        把 agent_steps（工具发现）拼进 request
              ▼
        现有流式 generate 出发言（§6.11 逐 token/人设/点账本全保）
```

- **β 关键**：planner 的 `final` 是「停止用工具」信号、**不是答案**；发言仍由现有 `generate` 产出（带工具发现 context）。
- **门控（A3）**：`turn` 未注入 `plan_stream`+`execute` = 纯发言（今天行为）。单机/pip 路径不依赖沙箱。
- **trace**：真用了工具 → 按 (speaker,turn) 存 `turn_traces` + 圆桌 SSE 实时冒 `tool_status`/`tool_call`/`tool_result` 子事件（前端进度行 + "查看执行"抽屉）。

### 9.2 执行子图的内部节点（非 registry 原语，orchestrator/app/nodes）

| 节点 | 职责 |
|---|---|
| `llm_plan` | 收 planner stream → 解析窄 JSON intent：`tool_call` 写 `pending_tools` / `final` 写 `output` |
| `tool_dispatch` | 消费队首 `pending_tools` → 经注入 `execute` 跑 → 成功/失败/降级都成 `ToolResult`，记 `agent_steps` |
| `loop_guard` | 纯路由：abort/done → pending tool → sandbox down → tool error → results → output → 再 plan |
| `plan_stream.default_plan_stream` | 真 planner：ReAct prompt（工具目录 + scratchpad）→ 下一个 tool_call/final |
| `plan_stream.default_tool_gate` | **准入门（S16b）**：廉价单一职责 yes/no「这轮需不需要工具」，保守默认 True |

### 9.3 工具面（三类，统一走 `mcp_call`/`sandbox_exec` 协议）

| 工具 | 来源 | 说明 |
|---|---|---|
| `sandbox_exec` | OpenSandbox（§6.23） | 隔离沙箱跑 shell/python（code-interpreter）；无沙箱时 prompt 不列 |
| MCP 工具 | `McpServer` 表 + `McpRegistry`（§6.24/S13f） | 多 server→一份目录 + 按 tool_name 路由；CRUD 后**热加载无需重启**（§6.26/S15） |
| 内置工具 | in-process（§6.25/S14） | `fetch_url` / `web_search`（DuckDuckGo 免 key）；作"in-process MCP server"塞进 registry，复用 mcp_call |

### 9.4 防空转（S16，§6.27）

工具化让纯聊问题也可能被 reasoning 模型反射性"先探一探"而空转调工具。两道防线：
- **S16a 防空转**：`agent_steps` scratchpad 渲真 `tool_name`+args（让 planner 看清已调过什么）；循环挡重复 intent；提示词加"不需工具直接 final"出口。
- **S16b 准入门（治本）**：`tool_gate` 在工具阶段前判「需不需要工具」，纯聊→跳过、根本不进 ReAct。`create_app` 从 `plan_model` 自动建（server.py 零改）；可注入快模型/假门（测试）。

### 9.5 接入点

- 圆桌/relay 的 `turn`（`build_roundtable_recipe(plan_stream, execute, tool_gate)`，create_app lifespan wire）。
- standalone `POST /execution/run`（SSE：trace/status/output + 心跳）——独立执行 loop，与圆桌 turn 共用一份 `ToolExecutorGate`/`SessionStore`（同 group_key 沙箱复用）。
- 执行子图用独立 durable checkpointer（与配方 saver 分离避 thread 冲突）。

## 10. 当前能力边界（设计新原语前必读）

1. **状态扁平、无子群/嵌套**：`GroupState` 只有一个 `roster`、一条 `history`、一本 `claims`。无法表达"成员分域、各域子群先开圆桌、结论再上提"。
2. **原语都在全 roster 上动**：`frame` 分维度（非分组）、`schedule` 从全员选一、`turn` 单人、`fanout` 全员并行。没有"对子集跑子配方再汇总"的原语。
3. **L3 产图词表只有 roundtable/fanout**：`RecipePlan` 无"分层/分阶段"选项，`assemble_recipe` 只裁两张扁平模板——故 AI 搭出来的总是单层圆桌或扇出。
4. **无递归/复合节点**：DAG 节点只能是 registry 里的原语，不能"一个节点本身是一张子图/子配方"。（`turn` 的执行子图 §9 是节点**内部硬接**的工具阶段、不可被配方层组合，不算反例——它对 DAG 仍是单个 `turn` 节点。）
5. **通用 runner 限 roundtable 形状**：web ChatPage 处理 framed/delta/turn/clarify/human_gate/deliver/output；含 `curate` 等其它人在环原语的自定义图需另做通用 runner。

> 要做**分层圆桌**（子群领域圆桌 → 跨域圆桌），需新增 `breakout` 类原语（roster 分域 + 子图 map + 汇总上提）+ 状态嵌套 + 扩 `RecipePlan`/`assemble` 词表——见后续切片设计。
