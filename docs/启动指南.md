# Chorus 启动指南

Chorus 由**三个进程**组成，按需启动：

| 进程 | 角色 | 默认地址 | 何时需要 |
|---|---|---|---|
| **orchestrator（大脑）** | LangGraph 编排服务，所有业务智能 | `127.0.0.1:8900` | **必须**（web / AstrBot 都连它） |
| **web（前端）** | Vue3 产品界面（圆桌/策展/配方/好友/模型/工具/历史） | `127.0.0.1:5173` | 本地用产品时 |
| **AstrBot（脸）+ group_relay 桥** | N 个 telegram bot 收发 + LLM 委托 | dashboard `:6185`、桥 `:9876` | 接 telegram 群，或好友用 `kind=astrbot` 后端 |
| **OpenSandbox server**（可选）| 隔离沙箱，让圆桌 AI 跑 Python/bash | `127.0.0.1:8080` | 要让 AI 执行代码时 |

> 只在本地 web 里跑圆桌/扇出/配方，**不接 telegram，也不用 AstrBot** —— 只起 orchestrator + web 即可。
> AstrBot 仅在「真接微信/telegram 群」或「某好友的 LLM 走 AstrBot 已配 provider（kind=astrbot）」时才需要。
> 执行层（沙箱/MCP 工具）默认关闭，不影响纯发言的圆桌/扇出，按需见 **§四**。

## 一、AstrBot 启动（脸 + 桥）

AstrBot 以 **vendored** 形式放在 `Chorus/astrbot/`（框架本身 .gitignore 不入库，仅 `data/plugins/group_relay/` 进 git）。`group_relay` 插件**就地**在 `astrbot/data/plugins/group_relay/`，AstrBot 启动时自动加载、无需复制/软链。

### 1. 环境（一次性）
AstrBot 需 **Python 3.10+**，用**独立环境**（不要和 orchestrator 的 `.venv` 混——本仓库用 conda env `astrbot_env`）：

```powershell
# 首次：建环境 + 装依赖
conda create -n astrbot_env python=3.11 -y
conda activate astrbot_env
cd E:\allwork\download\agent\Chorus\astrbot
pip install -r requirements.txt
```

### 2. 启动
```powershell
cd E:\allwork\download\agent\Chorus\astrbot
E:\AnacondaEnvs\astrbot_env\python.exe main.py
```
- 首次启动会下载 WebUI（dashboard）静态文件；失败可按日志提示手动下 `dist.zip` 解压到 `astrbot/data/`。
- 启动后：
  - **dashboard** 在 `http://127.0.0.1:6185`（默认用户 `astrbot`，首次登录设密码）。
  - **group_relay 桥** 自动在 `127.0.0.1:9876` 起（出站 `/outbound`、LLM 委托 `/llm`）；日志可见「group_relay 桥已起」。
- 数据根 = `astrbot/data/`（配置在 `data/cmd_config.json`，**含 token/密钥，不入库**）。

### 3. 配 telegram bot（接群时才需要，手动）
1. **BotFather** 为每个 AI 身份建一个 telegram bot，拿各自 token；对每个 bot 发 `/setprivacy` → **Disable**（否则收不到群里别人的消息）。
2. dashboard → 平台（platform）→ 加 N 个 telegram 实例（各填 token），每个实例有唯一 **id**（即 `bot_id`）；都置 RUNNING。
3. 在 **Chorus web 的「好友」页**给每个 Contact 填 `bot_ref` = 对应 telegram 实例 id（大脑出站据此路由「该好友发言→对应 bot」）。

### 4. group_relay 桥配置（可选）
插件配置（dashboard 插件页，或 `data/plugins/group_relay/_conf_schema.json` 的默认）：
- `bridge_port`：出站桥端口，默认 `9876`。
- `brain_url`：大脑地址，默认 `http://127.0.0.1:8900`（入站群消息转发到 `{brain_url}/relay/inbound`）。

> 桥同时提供 `POST /llm`（S7.1e）：好友的 LLM 后端选 `kind=astrbot` 时，大脑把发言委托给 AstrBot 里 `provider_id` 对应、**已配好已测过**的 provider。此时 AstrBot 必须在跑。

## 二、orchestrator 启动（大脑，必须）

### 1. 环境（一次性）
> 用 orchestrator 自己的 `.venv`；系统 Python 3.14 缺依赖，务必用 venv。

```powershell
cd E:\allwork\download\agent\Chorus\orchestrator
python -m venv .venv
.venv\Scripts\python.exe -m pip install -e .
```

### 2. LLM 配置（全局默认模型）
复制 `orchestrator/.env.example` 为 `orchestrator/.env`（不入库），填任意 OpenAI 兼容后端：
```
LLM_BASE_URL=https://your-openai-compatible-endpoint/v1
LLM_API_KEY=sk-xxxx
LLM_MODEL=kimi-k2.6
LLM_TEMPERATURE=0.75
```
> 这是**全局默认模型**（主持人/澄清/汇总，及未单独绑后端的好友都用它）。
> 想让 **ada1=gpt、ada2=deepseek** 这种「每好友独立模型」，到 web 的**「模型」页**建 LLM 后端（直接粘贴 key），再到「好友」页给好友绑定——见 §6.18。

### 3. 启动
```powershell
cd E:\allwork\download\agent\Chorus\orchestrator
.venv\Scripts\python.exe -m app.server      # 监听 127.0.0.1:8900
```
健康检查：浏览器开 `http://127.0.0.1:8900/health` 应返回 200。

> **执行层（沙箱 / MCP 工具）默认关**：纯发言的圆桌/扇出不需要它。要让圆桌里每个 AI 发言前先「跑代码 / 调工具」，按 **§四** 配环境变量再启动。

## 四、执行层：沙箱 + MCP 工具（可选，§6.25）

圆桌里每个 AI 发言前可先「用工具干活」（跑代码 / 调 MCP / 搜网页），再基于结果发言。不需要此功能可完全跳过本节。

**三层工具面，从易到难按需叠加：**

| 层 | 工具 | 何时生效 |
|---|---|---|
| **内置（开箱即用）** | `fetch_url` / `web_search`（DuckDuckGo，免 key） | 默认就有，orchestrator 起来即可用 |
| **MCP** | 任意 MCP server（filesystem / git / 自定义…） | `.env` 加 `CHORUS_EXECUTION=1`，然后在 web「工具」页注册 |
| **沙箱** | Python / bash 代码隔离执行 | 先起外部 OpenSandbox server，再填 `CHORUS_SANDBOX_DOMAIN` |

---

### 4.1 内置工具（无需配置）

默认启用，无需任何配置。orchestrator 启动后圆桌 AI 即可调用：
- `fetch_url`：抓取指定 URL 正文（httpx GET）
- `web_search`：DuckDuckGo 搜索（免 API key）

---

### 4.2 MCP 工具

在 `orchestrator/.env` 加一行：

```
CHORUS_EXECUTION=1
```

重启 orchestrator，然后打开 web → 导航「工具」→ 「从预设添加」或手填注册 MCP server。

**表单字段说明：**

| 字段 | 说明 |
|---|---|
| transport | `stdio`（起本地子进程）或 `sse`（连远程 HTTP SSE server） |
| command | **stdio 专用**：可执行文件，如 `npx`、`uvx`、`python` |
| args | **stdio 专用**：参数按空格分隔填入，整体会被拆成数组传给子进程 |
| url | **sse 专用**：远程 server 的 SSE 端点，如 `http://127.0.0.1:3000/sse` |

**官方预设（点「从预设添加」一键预填）：**

| 预设 | command | args（★ 需改为实际路径） | 前置依赖 |
|---|---|---|---|
| filesystem | `npx` | `-y @modelcontextprotocol/server-filesystem ★/你的目录` | node / npx |
| fetch | `uvx` | `mcp-server-fetch` | uv / uvx |
| git | `uvx` | `mcp-server-git --repository ★/你的仓库路径` | uv / uvx |
| memory | `npx` | `-y @modelcontextprotocol/server-memory` | node / npx |
| sequential-thinking | `npx` | `-y @modelcontextprotocol/server-sequential-thinking` | node / npx |
| time | `uvx` | `mcp-server-time` | uv / uvx |

> **注意**：args 框按空格拆数组，**带空格的路径暂不支持**（改用 sse 或后续版本增强）。
> 在 web 里新增 / 修改 MCP server 后，需**重启 orchestrator** 才生效（工具目录在启动时一次性 refresh）。

### 4.3 沙箱（OpenSandbox server）

沙箱允许圆桌 AI 在隔离容器里跑 Python / bash，结果安全可控。需要先起一个外部 OpenSandbox server（依赖 Docker）。

#### 起 OpenSandbox server

> 项目地址：https://github.com/opensandbox-group/OpenSandbox
> 前置：Python 3.10+，Docker（本地执行必须）

```bash
# 1. 安装
uv pip install opensandbox-server
# 或：pip install opensandbox-server

# 2. 生成 Docker 示例配置
opensandbox-server init-config ~/.sandbox.toml --example docker

# 3. 编辑 ~/.sandbox.toml（关键字段）
```

最小 TOML（Docker 模式）：

```toml
[server]
host = "0.0.0.0"
port = 8080
api_key = "your-secret-api-key"   # 强烈建议设置；留空则须 OPENSANDBOX_INSECURE_SERVER=YES

[runtime]
type = "docker"
execd_image = "..."               # 见 OpenSandbox 文档获取推荐镜像名
```

```bash
# 4. 启动
opensandbox-server
C:\Users\Lenovo\AppData\Roaming\Python\Python314\Scripts\opensandbox-server.exe --config ~/.sandbox.toml
# 指定配置：opensandbox-server --config /path/to/sandbox.toml
# api_key 为空时（非生产）需加前置变量：OPENSANDBOX_INSECURE_SERVER=YES opensandbox-server

# 5. 健康检查
curl http://127.0.0.1:8080/health    # 返回 200 即正常
# API 文档：http://127.0.0.1:8080/docs
```

#### Chorus 侧配置

在 `orchestrator/.env` 加：

```
CHORUS_SANDBOX_DOMAIN=127.0.0.1:8080          # host:port，不含 http://
CHORUS_SANDBOX_API_KEY=your-secret-api-key    # 与 sandbox.toml [server].api_key 一致
```

重启 orchestrator，启动日志出现类似 `execution enabled — sandbox: 127.0.0.1:8080, MCP: from DB registry` 即表示沙箱 + MCP 全部接入。

**执行层 env var 速查：**

| 变量 | 含义 | 示例 |
|---|---|---|
| `CHORUS_SANDBOX_DOMAIN` | OpenSandbox server 的 `host:port` | `127.0.0.1:8080` |
| `CHORUS_SANDBOX_API_KEY` | API Key，与 TOML `[server].api_key` 一致 | `your-secret-api-key` |
| `CHORUS_EXECUTION=1` | 仅启用 MCP，不接沙箱 | `1` |

组合规则：
- `CHORUS_SANDBOX_DOMAIN` 已填 → 沙箱 + MCP 全开（`CHORUS_EXECUTION=1` 可省）
- 只有 `CHORUS_EXECUTION=1`，无 `CHORUS_SANDBOX_DOMAIN` → 仅 MCP，无沙箱
- 两者都没有 → 纯发言（默认，现有测试/产品路径不依赖工具）

## 启动顺序与最小组合

- **只在本地用产品（不接群）**：① orchestrator → ② web。打开 `:5173` 即用。
- **接 telegram 群 / 用 kind=astrbot 后端**：① orchestrator → ② AstrBot（含桥）→ ③ web。
  - 先起大脑（桥入站要连 `:8900`）；AstrBot 后起即可，桥起来就会连大脑。
- **圆桌 AI 需用工具（沙箱）**：① OpenSandbox server → ② orchestrator（填好 env）→ ③ web。

## 端口速查

| 服务 | 地址 |
|---|---|
| orchestrator 大脑 | `127.0.0.1:8900`（`/health`） |
| web 前端（dev） | `127.0.0.1:5173` |
| AstrBot dashboard | `127.0.0.1:6185` |

## 常见问题

- **大脑起不来 / 报缺依赖**：确认用的是 `orchestrator/.venv` 而非系统 Python（系统 3.14 缺 pytest-asyncio / langgraph-checkpoint-sqlite 等）。
- **改了模型表后报 `no such column` / `NOT NULL constraint`**：重启 orchestrator 即可——启动时自动迁移（补缺列 + 删改名残留旧列）。
- **bot 收不到群里别人消息**：BotFather 对该 bot `/setprivacy` 没设成 Disable。
- **好友 `kind=astrbot` 发言报错**：AstrBot 没在跑、或 `provider_id` 在 AstrBot 里不存在；桥 `/llm` 找不到 provider 会 404。
- **MCP 工具没出现在圆桌 AI 选项里**：在 web「工具」页增/改 server 后需重启 orchestrator（catalog 启动时一次性 refresh）。
- **沙箱连不上（execution disabled）**：检查 `CHORUS_SANDBOX_DOMAIN` 格式（`host:port`，不含 `http://`）；确认 OpenSandbox server 在跑（`curl http://127.0.0.1:8080/health`）；`CHORUS_SANDBOX_API_KEY` 与 TOML 一致。
- **OpenSandbox 启动要求确认 / 报 api_key 为空**：非生产环境可加 `OPENSANDBOX_INSECURE_SERVER=YES` 绕过，但生产建议设 `[server].api_key`。
- **token/密钥安全**：AstrBot 的 `data/cmd_config.json`、orchestrator 的 `.env`、注册表 `*.sqlite` 都已 gitignore，不入库。
