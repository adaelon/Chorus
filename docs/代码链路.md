## 2026-06-01 · S1.1 服务骨架 + GroupState + checkpointer

**触达**：
- `orchestrator/app/state.py:GroupState` — 新增共享状态(group_key/roster/history/turns_since_human/max_turns_per_human/pending_human)，含 `AgentSlot`/`Msg`
- `orchestrator/app/graph.py:build_app` — 空图(`noop` 节点)编译 + 挂 SqliteSaver
- `orchestrator/app/graph.py:make_checkpointer` — 裸 sqlite3 连接的文件 checkpointer（便于测试控制连接生命周期）
- `orchestrator/app/__main__.py:main` — 最小"服务进程"：invoke + reload 打印

**入口**：`python -m app`（HTTP API 留到 S1.6）。
**测试**：`tests/test_persistence.py::test_state_persists_across_restart` — 空图 invoke → 落 sqlite → 模拟重启 reload 出同一 state，并验证可水化成完整 `GroupState`。
**坑记**：LangGraph 只持久化**被写入的** channel，pydantic 默认字段不进 checkpoint（设计如此）→ 重启后靠 `GroupState(**snap.values)` 补默认。
**环境**：Python 3.14.4 + 独立 venv；langgraph 1.2.2 / langgraph-checkpoint-sqlite 3.1.0 / pydantic 2.13.4 / pytest 9.0.3。

## 2026-06-01 · S1.2 LLM 客户端封装 + retry

**触达**：
- `orchestrator/app/config.py:load_llm_settings` — 加载 LLM 配置，**复用 talk-agent/.env**（`CHORUS_DOTENV` 可覆盖；生产应注入真实环境变量）
- `orchestrator/app/llm.py:make_chat_model` — 造 `ChatOpenAI`（OpenAI 兼容 /v1 后端）
- `orchestrator/app/llm.py:robust_invoke` — tenacity retry 包 `model.invoke`，`RETRYABLE`=连接/超时/5xx 瞬时错误

**入口**：供后续节点（S1.3+）调用；本切片暂无独立 CLI/HTTP。
**测试**：`tests/test_llm.py` — `test_retry_recovers_from_transient_disconnect` / `test_retry_gives_up_and_reraises`（离线 mock 桩）+ `test_smoke_real_backend`（需 `CHORUS_RUN_SMOKE=1`，实测 9.5s 通过）。
**环境**：langchain-openai 1.2.2 / tenacity 9.1.4 / python-dotenv 1.2.2 / openai 2.38.0；后端 opencode zen go（deepseek-v4-pro）。
**注**：4xx 业务错误不重试，只重试瞬时错误（技术方案 §8 / 需求 §9.8 实测的偶发 APIConnectionError）。

## 2026-06-01 · S1.3 FANOUT 节点（并行生成）

**触达**：
- `orchestrator/app/state.py:Candidate` — 新增候选模型；`GroupState` 加 `candidates` 字段
- `orchestrator/app/llm.py:robust_ainvoke` — `robust_invoke` 的异步版（`ainvoke` + tenacity）
- `orchestrator/app/nodes/fanout.py:fanout` — `asyncio.gather` 并行 N 个 agent 产候选；占位 prompt（不拼人设）；`generate` 可注入便于测试

**入口**：供扇出配方调用（图组装见 S1.6）；直接 `await fanout(state)`。
**测试**：`tests/test_fanout.py` — 并行+计数（断言 `elapsed < 2×单次延迟`）+ 无 pending 时从历史取需求；用注入假 generate，不碰真实 LLM。
**注**：真实 LLM 路径走 `_default_generator(make_chat_model())` + `robust_ainvoke`；人设注入留 S2.2。
**环境**：pytest-asyncio 1.4.0（`asyncio_mode = "auto"`）。

## 2026-06-01 · S1.4 FRAME 节点 + CLARIFY 占位

**触达**：
- `orchestrator/app/nodes/_common.py:request_text` — 从 S1.3 fanout 提取的共享 helper（取触发需求）；`fanout.py` 改为复用它
- `orchestrator/app/nodes/frame.py:frame` — 主持人给 roster 每人分 `dimension`，返回更新后的 roster；`assign` 可注入
- `orchestrator/app/nodes/frame.py:_default_assigner` — 真实主持人：纯文本 JSON + `_extract_json_obj` 自解析 + `_Dimensions` 校验
- `orchestrator/app/nodes/clarify.py:clarify` — 直通占位（恒"信心够"），真实信心自评留 S3.5

**入口**：供配方调用（图组装见 S1.6）；`await frame(state)` / `await clarify(state)`。
**测试**：`tests/test_frame.py` — FRAME 每 slot 非空 dimension（离线假 assign）+ CLARIFY 直通 + `test_frame_smoke_real_moderator`（`CHORUS_RUN_SMOKE=1`，实测 34s 通过）。
**坑记（重要）**：后端 `deepseek-v4-pro` 是**思考模型**，**不支持 `response_format`（json_schema）也不支持强制 `tool_choice`**——`with_structured_output` 两种方式都报 400。结构化输出一律改走**纯文本 JSON + 自解析**。后续 SCHEDULE(S3.2)/CURATE 同此约束。

## 2026-06-01 · S1.4b 结构化输出策略抽象（§6.9）

**触达**：
- `orchestrator/app/structured.py:structured_invoke` — 三策略（json_schema / function_calling / text_json），method 由配置定，text_json 通用兜底
- `orchestrator/app/structured.py:_extract_json_obj` / `_via_text_json` — 纯文本 JSON 解析 + schema 校验（从 frame 迁出、泛化）
- `orchestrator/app/config.py:LLMSettings.structured_method` — 新增配置（`LLM_STRUCTURED_METHOD`，默认 `text_json`）
- `orchestrator/app/nodes/frame.py:_default_assigner` — 改用 `structured_invoke`，删掉内联 JSON 解析

**入口**：`structured_invoke(model, msgs, schema)`；S3.2 SCHEDULE 复用。
**测试**：`tests/test_structured.py` — text_json 解析干净/带噪输出 + schema 指令注入（离线 mock）；frame 离线 + 真实 smoke 经新抽象仍通过。
**注**：json_schema / function_calling 分支用 langchain 现成封装，但**未对支持它们的真实模型 smoke 过**（当前后端不支持）；接第二个模型时配 `LLM_STRUCTURED_METHOD` 并 smoke（避免过拟合 deepseek）。

## 2026-06-01 · S1.5 CURATE 节点（pick/eliminate/reassign）

**触达**：
- `orchestrator/app/nodes/generate.py:default_generator` / `placeholder_messages` — 从 fanout 提取的共享候选生成；`fanout.py` 改为复用（FANOUT 与 CURATE.reassign 同源）
- `orchestrator/app/state.py:GroupState.picked` — 新增字段（人工策展选中的点/候选）
- `orchestrator/app/nodes/curate.py:curate` — apply `pick`/`eliminate`/`reassign`；含 `Pick`/`Eliminate`/`Reassign` 指令模型

**入口**：`await curate(state, commands)`；指令来自人/前端（结构化输入，无需 LLM 解析）。
**测试**：`tests/test_curate.py` — pick 进 `picked` / eliminate 移除候选 / reassign 产出含 point 的新候选（离线假 generate）。
**注**：信誉软加权（§10.5）+ eliminate 可逆性留 S2.3；本刀 eliminate 仅本场从 candidates 移除。
