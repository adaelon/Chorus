# 高维事件圆桌 · 技术方案与代码编排

> 本文讲清这个项目「是什么、怎么编排、核心逻辑在哪、为什么这么设计」。
> 配套阅读：[后端排查记录-opencode-go.md](后端排查记录-opencode-go.md)（讲清连后端时踩过的 4 个坑及原理）。

---

## 1. 一句话定位

给定一段**事件 / 材料**，由多个 LLM agent 组成一场**圆桌讨论**：先由「策划」读懂事件、定出核心问题、**按这件事临时挑出 3~5 个真正相关的分析视角**；各视角依次陈述、交叉质询；最后由「主笔」给出一个**外行能看懂、且尽量贴近现实的明确结论**。

提供两种入口：命令行（`main.py`）和网页（`web.py` + `static/index.html`），**共用同一套编排核心**（`roundtable.py`）。

---

## 2. 设计哲学（为什么这么做）

这套设计是被一次真实反馈逼出来的。最初的版本用一套**写死的固定维度**（时间线 / 利益方 / 制度 / 权力 / 叙事 / 反事实……）去套每一个事件，结果是：

> 当维度和用户的问题不相符时，根本没人在真的讨论这个问题，而是从那个维度**牵强附会**地硬凑。

于是定下三条原则，贯穿全部代码：

1. **视角按事件动态生成，不套模板。** 不相关的角度根本不上场，从根上消除牵强附会。
2. **有料才深入，没料就直说"不适用"。** 每个 agent 的准则里都明确允许"从这个角度看，这里信息有限 / 不是关键"，不为显得有深度而硬凑。
3. **要给结论，不和稀泥。** 不再罗列"也可能 A 也可能 B"，证据支持哪个判断就明确说出来，同时保留**真正可能动摇结论**的少数关键存疑。

目标只有一句：**得出更符合现实的结论，并且让用户看懂。**

## 5. 核心编排：`RoundTable`

### 5.1 三个固定角色 + N 个动态视角

| 角色 | 名称 | 何时存在 | 职责 |
|---|---|---|---|
| 策划 | `PLANNER_NAME` | 一直 | 读事件 → 定核心问题 → **动态挑视角** |
| 主持人 | `MODERATOR_NAME` | 一直 | 每轮挑出最影响结论的张力，逼正面交锋；不站队、不下结论 |
| 主笔 | `SYNTHESIZER_NAME` | 一直 | 综合全场 → 给明确结论（带信心与存疑） |
| 视角持有者 | 由策划命名 | **运行时动态创建** | 死守自己这一视角，扣住核心问题发言 |

三个固定角色在 `__post_init__` 里就建好；视角 agent 要等策划产出方案后，在 `_phase_plan` 里**动态创建**。

### 5.2 四阶段流程

```
run(event)
  │
  ├─ _phase_plan      策划读事件 → JSON{framing, core_question, perspectives[]}
  │                   → 解析 → 动态注册视角 agent → 发出 plan 卡片
  │
  ├─ _phase_opening   每个视角围绕核心问题首轮陈述（能看到本轮前面的发言）
  │
  ├─ _phase_debate_round × (rounds-1)
  │     ├─ 主持人挑 2~3 个张力点（moderate）
  │     └─ 各视角针对张力正面回应（debate）
  │
  └─ _phase_synthesis 主笔综合全场 → 结论（synthesis）
```

每一步产出一个 `Turn(speaker, icon, text, phase)`，经 `_emit()` 同时**追加进 transcript** 并**触发 `on_turn` 回调**——这是连接编排与展示层的关键接缝。

### 5.3 「共享黑板」式状态管理

每个 agent 调用都是**无状态**的：不依赖 SDK 的会话记忆，而是 `RoundTable` 自己维护 `transcript`，每次把「事件 + 核心问题 + 筛选后的已有发言」拼成完整 prompt 喂进去。

```python
def _transcript_text(self, phases=None) -> str:
    # 按 phase 过滤已有发言，精确控制"谁在这一步能看到什么"
```

好处：完全掌控上下文边界（陈述阶段只看陈述、质询阶段看陈述+质询、主笔看全部），便于调试，也不受后端会话能力差异影响。

## 6. 三个关键技术决策（核心逻辑所在）

这三点是项目能跑起来、且产出可用的真正原因，全部源自 [排查记录](后端排查记录-opencode-go.md)。

### 6.1 人设折叠进 user 消息，不发 system

部分兼容网关（如 opencode go）**拒绝 system 角色**消息，直接 400 `System messages are not allowed`。而 `as_agent(instructions=...)` 正是把 instructions 变成 system 消息。

**对策**：建 agent 时**不传 `instructions=`**；改用一张 `self._persona: dict[name -> 人设]` 表，在 `_ask()` 里按 `agent.name` 查表，把人设折叠进 user 提示：

```python
async def _ask(self, agent, prompt):
    persona = self._persona.get(agent.name, "")
    full = (f"【你的角色设定】\n{persona}\n\n——以下是本轮任务——\n\n{prompt}"
            if persona else prompt)
    ...
```

动态视角在 `_register_perspectives` 里同样把 instructions 写进这张表，与固定角色走完全一致的通道。

### 6.2 一律走流式（stream=True）

deepseek 这类**推理模型**在吐首字节前要"思考"数十秒（实测 ~48s）。**非流式**请求在这段静默期会被本地代理 / 网关当成空闲连接掐断，报 `Server disconnected without sending a response`。

**对策**：`_ask()` 统一用 `agent.run(full, stream=True)`，边到边收 `upd.text` 拼接。流式保持连接活跃，绕开空闲断连。

```python
parts = []
async for upd in agent.run(full, stream=True):
    if getattr(upd, "text", None):
        parts.append(upd.text)
return "".join(parts).strip()
```

### 6.3 用 ChatCompletion 客户端 + base_url 收在 /v1

- **必须用 `OpenAIChatCompletionClient`**（走 `/chat/completions`），**不能用 `OpenAIChatClient`**——后者默认走 OpenAI 新版 `/responses` 端点，兼容网关多半没有，直接 404。
- **`base_url` 收在 `/v1`**，路径由 SDK 自动补。写成 `.../v1/chat/completions` 反而会拼成 `.../v1/chat/completions/chat/completions`。

> 这两点连同 6.1/6.2，构成接通后端的"四件套"，详见排查记录。

## 7. 策划产出的解析（动态视角的实现细节）

策划被要求**只输出一个 JSON 对象**：

```json
{
  "framing": "中立复述，分清事实与说法",
  "core_question": "本场要回答的核心问题",
  "perspectives": [
    {"name": "视角名", "icon": "🏪", "why": "为什么对这件事关键", "focus": "重点看什么"}
  ]
}
```

但 LLM 常会包代码块或加前后噪声，故 `_parse_plan` 做了容错：

1. 剥掉 ```` ```lang ```` 代码块包裹；
2. 取**最外层** `{ ... }`（`find('{')` 到 `rfind('}')`）；
3. `json.loads`，**失败返回 `{}`**。

随后 `_perspectives_from_plan` 把每个视角对象转成一个 `Dimension`（`key=f"p{i}"`），并把 `PERSPECTIVE_BASE` 准则 + 该视角的 name/why/focus 拼成它的 instructions。

**兜底链**：解析失败或没拿到视角 → 退回 `resolve_dimensions(None)` 的默认固定维度集，保证流程不中断。
**手动覆盖**：若构造 `RoundTable` 时传了 `dimensions`（CLI `--dims` / Web 选了视角），则跳过动态生成，直接用固定维度。

## 8. 展示层

### 8.1 同一回调，两种渲染

`RoundTable(on_turn=...)` 是唯一接缝：

- **CLI**（`main.py`）：`on_turn=_render_turn`，按 `phase` 配色用 `rich.Panel` 实时打印；synthesis 阶段用 `rich.Markdown` 渲染。
- **Web**（`web.py`）：`on_turn=queue.put_nowait` 把 `Turn` 塞进 `asyncio.Queue`；`generate()` 一边 `await table.run()`，一边把队列里的发言**逐条转 NDJSON** 流式 `yield` 给前端。

```python
while not (task.done() and queue.empty()):
    turn = await asyncio.wait_for(queue.get(), timeout=0.25)  # 边跑边推
    yield _ndjson({"type": "turn", ...})
```

消息类型：`meta`（开场，含 `auto` 标志=是否动态选角）→ 多条 `turn` → `done`（存档路径）或 `error`。

### 8.2 前端 Markdown + LaTeX 渲染

`static/index.html` 自写极简渲染器 `renderMd`，无重型依赖，仅 KaTeX 走 CDN。**渲染顺序是关键**：

```
原文
 │ ① 抽出数学公式 $…$ / $$…$$ / \(…\) / \[…\] → 占位符（私有区字符 … 包裹）
 │ ② escapeHtml + 标题/加粗/代码/有序无序列表/段落
 │ ③ 用 KaTeX 把占位符还原成公式 HTML（失败则退回原始 $…$ 文本）
HTML
```

先抽公式再做 markdown，是为了避免 `**`、`` ` `` 等行内规则破坏 LaTeX；占位符用私有区字符包裹，确保不会误吞正文里的数字（如 "3~5"、"40%"）。行内 `$…$` 要求紧贴非空白，避开把 "$5 和 $10" 当公式。

> 对应地，`dimensions.py / roundtable.py` 的准则里加了一条：**涉及定量关系才用 LaTeX，纯文字能说清就别硬套**——与"不牵强附会"原则一致。

### 8.3 存档

`save_markdown` 被 CLI 与 Web 共用，把「事件原文 + 全程发言 + 结论」写成 `outputs/roundtable-时间戳.md`，保留原始 LaTeX，可在任何支持数学的 Markdown 阅读器里查看。

## 10. 端到端数据流（全景）

```
事件文本
  │
  ▼
[策划] ──JSON──→ _parse_plan ──→ 核心问题 + 动态视角[] ──→ 动态注册 agent
  │                                                          │
  │  (emit: plan)                                            ▼
  ▼                                              [视角1..N] 首轮陈述 (emit: open)
[主持人] 挑张力 (emit: moderate)                              │
  │                                                          ▼
  └────────→ [视角1..N] 交叉质询 (emit: debate) ←── 循环 rounds-1 轮
                         │
                         ▼
                    [主笔] 综合 → 明确结论 (emit: synthesis)
                         │
   每个 emit ──→ on_turn 回调 ──┬─→ CLI: rich 实时面板
                                └─→ Web: Queue → NDJSON 流 → 前端 renderMd
                         │
                         ▼
                save_markdown → outputs/*.md
```

## 11. 扩展点

| 想做什么 | 改哪里 |
|---|---|
| 换后端 / 模型 | 只改 `.env`（`config.py` 是唯一出口） |
| 调整角色人设 | `roundtable.py` 的 `*_INSTRUCTIONS` 常量 |
| 改动态选角规则 | `PLANNER_INSTRUCTIONS` + `_perspectives_from_plan` |
| 加固定维度（手动模式用） | `dimensions.py` 的 `DIMENSIONS` 列表 |
| 改讨论轮数 / 视角数上限 | `RoundTable(debate_rounds=, max_perspectives=)` |
| 前端样式 / 渲染能力 | `static/index.html` |

核心不变量：**新增任何展示层，只需消费 `on_turn` 回调；新增任何后端，只需是 OpenAI 兼容 + 收在 `/v1`。**
